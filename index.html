Chapter 1 — Foundations
01 Data vs Information vs Database ▶

Data is raw facts — just numbers, words, or symbols with no meaning by themselves. Information is when you give data meaning and context. A Database is an organised collection of data stored so it can be accessed, managed, and updated easily.

Think of a school: "85, 90, 78" is just data. "Rahul scored 85 in Maths, 90 in Science, 78 in English" is information. The school's complete marks record of all students, stored in a system — that's a database.
ConceptDefinitionExample
DataRaw, unprocessed facts42, "John", 2006-01-15
InformationProcessed data with meaning"John is 18 years old, born Jan 15"
DatabaseOrganised collection of related dataSchool's student records system
02 DBMS vs File System ▶

A File System is the old-school way of storing data — just files on your computer like .txt or .csv. A DBMS (Database Management System) is software that makes storing, retrieving, and managing data much smarter, safer, and easier.

File System = keeping all your notes as loose papers stuffed in a drawer. DBMS = a well-organised filing cabinet with labeled folders, where you can find anything in seconds, and two people can access different files at the same time without messing things up.

Concurrent AccessPoor supportExcellent (locking, transactions)

Query SupportNone (must write code)SQL queries

04 3-Tier Architecture (External, Conceptual, Internal) ▶

DBMS uses a 3-level architecture (also called ANSI/SPARC model) to separate what users see from how data is actually stored.

Think of a restaurant: the menu (what customers see) is the external level. The recipes and kitchen workflow is the conceptual level. The actual storage of ingredients in the pantry is the internal level. Customers don't need to know how the pantry is organised!
LevelAlso CalledWhat it represents
External LevelView Level / User LevelWhat individual users see — their customised view of the data
Conceptual LevelLogical LevelThe complete logical structure of the whole database (tables, relationships) — no physical storage details
Internal LevelPhysical LevelHow data is actually stored on disk — indexes, file formats, storage blocks

This separation allows changes at one level without affecting others — that's called Data Independence.

05 Schema vs Instance ▶

Schema is the blueprint/design of the database — the structure. It defines what tables exist, what columns they have, what types those columns are. It rarely changes.
Instance is the actual data in the database at a specific moment in time. It changes every time you insert, update, or delete data.

Schema = the blueprint of a house (rooms, doors, walls). Instance = the actual furniture, people, and stuff inside the house right now. The blueprint stays the same, but what's inside keeps changing.
SchemaInstance
What is it?Structure / designActual data at a point in time
Changes?RarelyFrequently
Also calledIntensionExtension
ExampleTable "Students" with columns (RollNo, Name, Age)Row: (101, "Priya", 16)
06 Data Independence (Logical & Physical) ▶

Data Independence means you can change one level of the database without having to change other levels. It's one of the biggest advantages of using a DBMS.

  • Physical Data Independence: You can change how data is stored on disk (e.g., switch from one file format to another, add indexes, move data to SSD) without changing the logical schema or user views. This is easier to achieve.
  • Logical Data Independence: You can change the logical/conceptual schema (e.g., add a new column to a table, merge two tables) without changing the external views that users see. This is harder to achieve.
Physical independence = change the filing cabinet brand without changing the folder system. Logical independence = reorganise the folder system without changing what staff see on their computer screens.
Chapter 2 — ER Model & Relational Model
07 ER Model (Entity-Relationship Model) ▶

The ER Model is a way to visually design a database before building it. It uses a diagram called an ER Diagram to show what data we'll store and how different pieces of data relate to each other.

It's like drawing a map of your app before coding it. If you're building a school app, you'd draw boxes for "Student", "Teacher", "Subject" and lines showing how they connect — "Student ENROLLED IN Subject".

Main Components

  • Entity → A real-world thing (e.g., Student, Book, Order)
  • Attribute → A property of an entity (e.g., Student has Name, Age, RollNo)
  • Relationship → Connection between entities (e.g., Student "TAKES" Course)

In an ER diagram: Rectangles = Entities, Ellipses = Attributes, Diamonds = Relationships, Lines = Connections.

  • Ternary: Between three entities (Doctor PRESCRIBES Medicine TO Patient)
09 Strong vs Weak Entity ▶

A Strong Entity has its own unique identifier (primary key) and can exist on its own.
A Weak Entity does NOT have its own unique key — it depends on another entity to be identified.

Strong entity = a person who has their own Aadhar card and can exist independently. Weak entity = a dependent child in an insurance policy — the child's record only makes sense in the context of the policyholder (parent).

10 Cardinality & Participation Constraints ▶

Cardinality tells us how many instances of one entity can be related to instances of another.

CardinalityMeaningExample
1:1 (One-to-One)One A relates to exactly one BOne person has one passport
1:N (One-to-Many)One A relates to many B'sOne teacher teaches many students
M:N (Many-to-Many)Many A's relate to many B'sStudents take many courses; courses have many students

Participation Constraints

This tells us whether every entity MUST participate in a relationship, or if it's optional.

  • Total Participation (mandatory): Every entity MUST be in the relationship. Shown as double line in ER diagram. Example: Every employee MUST work in a department.
  • Partial Participation (optional): Some entities may not participate. Shown as single line. Example: Some employees MAY manage a project (not all managers).
11 ER to Relational Mapping ▶

Once we design an ER diagram, we convert it into actual database tables. Here are the rules:

  • Strong Entity → becomes a table. Its attributes become columns. Primary key becomes the table's PK.
  • Weak Entity → becomes a table with its partial key + the owner's PK as combined primary key.
  • 1:1 Relationship → Add the PK of one entity as a Foreign Key in the other.
  • 1:N Relationship → Add the PK of the "1-side" entity as a FK in the "N-side" table.
  • M:N Relationship → Create a new junction/bridge table with the PKs of both entities.
  • Multi-valued Attribute → Create a separate table for it.
M:N is like a "friend of a friend" situation. Students ↔ Courses: we create an "Enrolment" table that links student IDs with course IDs.

17 Joins — Inner, Left, Right, Full Outer, Cross, Self ▶

Joins combine rows from two or more tables based on a related column. This is how we pull together linked data.

-- INNER JOIN: Show student names with their course names
SELECT S.Name, C.CourseName
FROM Students S
INNER JOIN Enrolment E ON S.RollNo = E.RollNo
INNER JOIN Courses C ON E.CourseID = C.CourseID;

-- SELF JOIN: Find employees and their managers (same table!)
SELECT E.Name AS Employee, M.Name AS Manager
FROM Employees E
LEFT JOIN Employees M ON E.ManagerID = M.EmpID;
INNER JOIN = only the Venn diagram overlap. LEFT JOIN = full left circle + the overlap. FULL OUTER = both full circles. CROSS JOIN = every student paired with every course — even if they're not enrolled.
18 Nested Queries / Subqueries ▶

A subquery is a query written inside another query. The inner query runs first and provides results to the outer query.

Like asking: "Give me a list of students who scored more than the average mark." You first need to find the average (inner query), then use it to filter students (outer query).
-- Find students who scored more than the average
SELECT Name, Marks
FROM Students
WHERE Marks > (SELECT AVG(Marks) FROM Students);

-- Find students enrolled in the 'DBMS' course
SELECT Name FROM Students
WHERE RollNo IN (
  SELECT RollNo FROM Enrolment
  WHERE CourseID = (SELECT CourseID FROM Courses WHERE CourseName = 'DBMS')
);

Subqueries can appear in: WHERE, FROM (as derived table), or SELECT clause.

19 Correlated Subqueries ▶

A correlated subquery is a subquery that references a column from the outer query. This means the inner query runs ONCE FOR EACH ROW of the outer query — so it's slower but more powerful.

Regular subquery = you calculate the class average once, then check every student. Correlated subquery = for each student, you check if their mark is higher than the average of their OWN specific class.
-- For each student, check if their marks are above their department's average
SELECT Name, Department, Marks
FROM Students S_outer
WHERE Marks > (
  SELECT AVG(Marks)
  FROM Students S_inner
  WHERE S_inner.Department = S_outer.Department  -- references outer query!
);

The inner query uses S_outer.Department — it changes for every row of the outer query. That's what makes it correlated.

21 NULL Handling in SQL ▶

NULL means "no value" or "unknown" — it's NOT the same as 0, empty string, or false. NULL is special and has tricky rules.

  • You CANNOT use = to check for NULL. Use IS NULL or IS NOT NULL
  • Any arithmetic with NULL gives NULL (e.g., 5 + NULL = NULL)
  • Any comparison with NULL gives UNKNOWN (e.g., NULL = NULL is NOT true — it's UNKNOWN!)
  • COUNT(*) counts all rows including NULLs, but COUNT(column) ignores NULLs
  • AVG, SUM, MIN, MAX all ignore NULLs
-- WRONG way to check NULL:
WHERE Email = NULL  -- This never works!

-- CORRECT way:
WHERE Email IS NULL
WHERE Email IS NOT NULL

-- COALESCE: Return first non-NULL value
SELECT Name, COALESCE(Phone, 'No Phone') AS Contact FROM Students;

-- NULLIF: Returns NULL if two values are equal
SELECT NULLIF(Score, 0) FROM Results; -- makes 0 become NULL
22 Views ▶

A View is a virtual table — it's not stored physically, but it shows the result of a saved SQL query. Every time you access the view, the query runs fresh.

Think of a view like a saved search on YouTube. It's not a new video — it's just a shortcut that runs a query and shows you the result. The actual videos (data) live in the real tables.
  • Simplicity: Hide complex queries behind a simple view name
  • Security: Show only certain columns to certain users (hide salary, hide passwords)
  • Reusability: Write a complex query once, reuse it everywhere as a view
-- Create a view showing only student names and marks (hiding personal info)
CREATE VIEW StudentMarks AS
SELECT Name, Department, Marks
FROM Students;

-- Use the view just like a table
SELECT * FROM StudentMarks WHERE Marks > 80;

-- Drop the view
DROP VIEW StudentMarks;

24 Triggers ▶

A Trigger is an automatic action that fires when something happens to a table — like INSERT, UPDATE, or DELETE. You don't call triggers — they fire automatically!

A trigger is like a motion sensor alarm. You don't manually activate it — it fires automatically when something moves (changes in the table).
  • BEFORE trigger: Runs BEFORE the operation (e.g., validate data before insert)
  • AFTER trigger: Runs AFTER the operation (e.g., log the change after update)
-- Trigger: Automatically log when a student record is deleted
CREATE TRIGGER LogDelete
AFTER DELETE ON Students
FOR EACH ROW
BEGIN
  INSERT INTO AuditLog (Action, RollNo, DeletedAt)
  VALUES ('DELETE', OLD.RollNo, NOW());
END;

Use cases: Audit logging, enforcing business rules, automatic calculations, sending alerts.

Chapter 4 — Indexes & Storage
25 Indexes — Dense, Sparse, Clustered, Non-Clustered ▶

An Index is like a book's index at the back — instead of reading every page to find "photosynthesis", you jump to page 87. Similarly, a database index helps find rows quickly without scanning every row.

Without an index = reading a 1000-page book page by page. With an index = flip to page 87 directly.
TypeDefinition
Dense IndexOne index entry for EVERY record in the table. Faster search but takes more space.
Sparse IndexIndex entries only for SOME records (usually first record per block). Less space, slightly slower.
Clustered IndexThe actual data rows are sorted and stored in the same order as the index. Only ONE per table. (Like a dictionary — words ARE in alphabetical order.)
Non-Clustered IndexA separate structure pointing to data rows. Data is NOT physically sorted by this index. Multiple allowed per table. (Like a book index pointing to page numbers.)
Primary IndexBuilt on a key field. Data file is ordered by the key.
Secondary IndexBuilt on a non-key field. Useful for queries on non-key columns.
26 B-Tree & B+ Tree Indexing ▶

Most databases use B+ Trees (B-plus trees) for indexing. They are tree-shaped data structures that keep data sorted and allow fast searching, insertion, and deletion.

Imagine a hierarchy: first you narrow it down to a continent, then a country, then a city, then a street, then a house. That's how a B+ Tree finds a value quickly — by navigating levels.
B-TreeB+ Tree
Data inAll nodes (internal + leaf)Only LEAF nodes
Leaf linkingNoYes — leaves are linked as a linked list
Range queriesSlowerVery fast (just traverse the leaf chain)
Used inLess commonMost RDBMS — MySQL, PostgreSQL, Oracle

B+ Trees guarantee O(log n) search time. All leaf nodes are at the same level (balanced). Internal nodes only store keys for navigation; actual data pointers are at leaves.

Chapter 5 — Functional Dependencies & Normalization
27 Functional Dependency ▶

A Functional Dependency (FD) means: if you know the value of attribute A, you can always determine the value of attribute B. Written as A → B (A determines B, or B is functionally dependent on A).

If you know a student's Roll Number, you can always find their Name. So: RollNo → Name. This is a functional dependency. But Name → RollNo is NOT an FD (two students can have the same name).
  • Trivial FD: A → B where B is a subset of A. Example: {RollNo, Name} → Name. Always true.
  • Non-trivial FD: A → B where B is NOT a subset of A. These are the meaningful ones.
  • Full FD: B depends on the ENTIRE set A, not just part of it.
  • Partial FD: B depends on only PART of a composite key A. (Problematic — leads to 2NF violation)
28 Armstrong's Axioms ▶

Armstrong's Axioms are rules that let you derive ALL possible functional dependencies from a given set. They are sound (only produce valid FDs) and complete (can generate all FDs).

AxiomRuleExample
ReflexivityIf B ⊆ A, then A → B{Name, Age} → Name (trivial)
AugmentationIf A → B, then AC → BCIf RollNo → Name, then {RollNo, Dept} → {Name, Dept}
TransitivityIf A → B and B → C, then A → CIf RollNo → DeptID and DeptID → DeptName, then RollNo → DeptName

Secondary Rules (derived from above):

  • Union: If A→B and A→C, then A→BC
  • Decomposition: If A→BC, then A→B and A→C
  • Pseudotransitivity: If A→B and CB→D, then CA→D
29 Closure of Attribute Set ▶

The closure of an attribute set X, written as X⁺, is the set of ALL attributes that can be determined from X using the given functional dependencies.

You start with some attributes. Keep applying the FD rules — every time you find a new attribute you can determine, add it to your set. Stop when you can't add anything new. That final set is the closure.

Example:

Given: FDs = {A→B, B→C, C→D}. Find closure of {A}.

Start: {A}
A → B  → add B:  {A, B}
B → C  → add C:  {A, B, C}
C → D  → add D:  {A, B, C, D}
No more rules to apply.

Closure of A = {A, B, C, D}

If the closure of a set = all attributes in the relation → that set is a super key!

30 Candidate Key Finding ▶

A Candidate Key is a minimal set of attributes whose closure = all attributes in the relation. To find all candidate keys:

  • Step 1: Find which attributes NEVER appear on the right side of any FD — they MUST be in every candidate key (they can't be derived from anything).
  • Step 2: Find which attributes appear ONLY on the right side — they can NEVER be part of a candidate key alone.
  • Step 3: Try combinations of the must-include attributes and check their closures.
  • Step 4: If closure = all attributes → it's a super key. If it's also minimal → it's a candidate key.
Think of it as finding the smallest "key ring" that opens all locks (determines all attributes). Remove even one key from the ring and you can't open all locks anymore.
31 Normalization — 1NF, 2NF, 3NF, BCNF ▶

Normalization is the process of organising a database to reduce redundancy (duplicate data) and ensure data integrity. We go through different "normal forms" — each one fixing a specific problem.

Normalization is like cleaning your messy room: 1NF = throw away things with no fixed place. 2NF = put each type of thing in its own drawer. 3NF = don't let drawers depend on other drawers — only on the room's master organiser. BCNF = stricter version of 3NF.
Normal FormRuleProblem Fixed
1NFNo multi-valued or repeating group attributes. Every cell must have ONE atomic value. Each row must be unique.Repeating groups, non-atomic values
2NFMust be in 1NF + No PARTIAL DEPENDENCY (non-key attribute must depend on the FULL primary key, not part of it)Partial dependencies — only matters if PK is composite
3NFMust be in 2NF + No TRANSITIVE DEPENDENCY (non-key attribute must not depend on another non-key attribute)Transitive dependencies
BCNFMust be in 3NF + For every FD X→Y, X must be a super keyCases where 3NF is still not clean enough

Most practical database design aims for 3NF or BCNF.

32 Partial Dependency ▶

A Partial Dependency occurs when a non-key attribute depends on only PART of a composite primary key (not the full key). This violates 2NF.

Imagine a table where PK is (StudentID, CourseID). If StudentName depends only on StudentID (not CourseID), that's a partial dependency — StudentName has nothing to do with CourseID!
-- BAD TABLE (has partial dependency):
Enrolment(StudentID, CourseID, StudentName, CourseName, Grade)
-- PK = (StudentID, CourseID)
-- StudentName depends only on StudentID → PARTIAL DEPENDENCY!
-- CourseName depends only on CourseID  → PARTIAL DEPENDENCY!

-- FIXED (2NF — split into three tables):
Student(StudentID, StudentName)
Course(CourseID, CourseName)
Enrolment(StudentID, CourseID, Grade)
33 Transitive Dependency ▶

A Transitive Dependency occurs when a non-key attribute A determines another non-key attribute B. So the key → A → B (indirectly through A). This violates 3NF.

-- BAD TABLE (has transitive dependency):
Student(StudentID, Name, DeptID, DeptName, DeptLocation)
-- PK = StudentID
-- StudentID → DeptID (fine — direct)
-- DeptID → DeptName (TRANSITIVE! DeptName depends on DeptID, not StudentID)
-- DeptID → DeptLocation (TRANSITIVE!)

-- FIXED (3NF):
Student(StudentID, Name, DeptID)
Department(DeptID, DeptName, DeptLocation)
Key → City → Country. You know the key gives you City. City gives you Country. Country is transitively dependent on Key — it should be in a separate table.
34 Multivalued Dependency & 4NF ▶

A Multivalued Dependency (MVD) occurs when one attribute in a table uniquely determines a set of values of another attribute, independently of all other attributes. Written as A →→ B.

A professor can teach multiple subjects AND has multiple phone numbers — these two facts (subjects, phone numbers) are completely independent of each other, but both depend on the professor. Storing them in one table causes redundant combinations.
-- BAD TABLE:
Professor(ProfName, Subject, Phone)
-- ProfName →→ Subject (multivalued)
-- ProfName →→ Phone (multivalued)
-- This forces every subject-phone combination to be repeated!

ProfName    Subject     Phone
Dr. Kumar   DBMS        9876543210
Dr. Kumar   DBMS        9123456789   ← duplicate subject just for new phone
Dr. Kumar   OS          9876543210   ← duplicate phone just for new subject
Dr. Kumar   OS          9123456789

-- FIXED (4NF — split into two tables):
ProfSubject(ProfName, Subject)
ProfPhone(ProfName, Phone)

4NF: Must be in BCNF + no non-trivial multivalued dependencies.

35 Lossless Join & Dependency Preserving Decomposition ▶

When we split (decompose) a table during normalisation, we need to make sure we don't lose information or lose the functional dependency rules.

  • Lossless Join Decomposition: When you split a table into smaller tables, you should be able to JOIN them back and get the EXACT original table — no extra or missing rows. This is the most important property.
  • Dependency Preserving Decomposition: All the original FDs should still be checkable within the new smaller tables, without needing to JOIN them back. If you must JOIN to check an FD, it's not dependency-preserving.
Lossless = cut a pizza and put it back together — you get the full pizza back, no piece is lost. Dependency preserving = every quality rule (FD) about the pizza can be checked without reassembling it — just by checking the individual slices.

Note: BCNF always gives lossless decomposition but may NOT preserve all dependencies. 3NF guarantees BOTH lossless join AND dependency preservation.

Chapter 6 — Transactions & Concurrency
36 Transactions ▶

A Transaction is a group of SQL operations that are treated as ONE unit of work. Either ALL operations succeed together, or NONE of them happen (rolled back).

Bank transfer: You transfer ₹1000 from Account A to Account B. That involves: (1) Deduct ₹1000 from A, (2) Add ₹1000 to B. If step 1 succeeds but step 2 fails (server crash), you'd lose ₹1000! Transactions ensure both steps happen together or neither happens.
START TRANSACTION;
  UPDATE Accounts SET Balance = Balance - 1000 WHERE AccNo = 'A101';
  UPDATE Accounts SET Balance = Balance + 1000 WHERE AccNo = 'B202';
COMMIT;  -- Make changes permanent

-- If anything goes wrong:
ROLLBACK;  -- Undo everything back to the start
37 ACID Properties ▶

ACID is what makes transactions reliable. Every good transaction must follow all four properties:

PropertyMeaningExample
AtomicityAll or nothing — either ALL operations of a transaction succeed, or NONE doIf deduction succeeds but credit fails → rollback both
ConsistencyDatabase must go from one valid state to another valid state — all rules/constraints must be satisfiedTotal money before transfer = total money after transfer
IsolationConcurrent transactions don't interfere with each other — each transaction runs as if it's aloneTwo people booking the last seat simultaneously won't both get it
DurabilityOnce a transaction is committed, changes are PERMANENT — even if the system crashesYour UPI payment confirmation means the money is gone for good
ACID is like a vending machine: it either gives you the snack AND takes the money (atomic), keeps inventory consistent, doesn't confuse two people's orders (isolated), and the snack is gone for good once dispensed (durable).
38 Transaction States ▶

A transaction goes through several states during its lifetime:

StateMeaning
ActiveTransaction is currently executing (reading/writing)
Partially CommittedLast operation has been executed, but changes are not yet written to disk
CommittedAll operations successful, changes are permanently saved to disk
FailedAn error occurred — transaction cannot proceed normally
AbortedTransaction has been rolled back — database restored to state before the transaction started
TerminatedTransaction has ended (either committed or aborted)
Think of an exam: Active = writing the exam. Partially committed = finished writing, waiting for paper to be collected. Committed = marks entered in system. Failed = caught cheating. Aborted = paper cancelled. Terminated = result declared.
39 Concurrency Control ▶

Concurrency Control manages multiple transactions running at the same time so they don't interfere with each other and cause incorrect results. Without it, we get problems:

ProblemWhat happensExample
Lost UpdateTwo transactions read the same value, both modify it — one update is overwritten by the otherTwo agents book the same seat; last one wins, first booking lost
Dirty ReadTransaction reads data that another uncommitted transaction wrote (which may later be rolled back)Reading a transferred amount before the bank confirms it
Non-Repeatable ReadA transaction reads a value twice but gets different results because another transaction changed it in betweenRe-reading a product price that changed during your checkout
Phantom ReadA query returns different rows on re-execution because another transaction inserted/deleted rows in betweenCounting seats, then getting a different count seconds later

Solutions include: Locking, Timestamp Ordering, Optimistic Concurrency Control, and MVCC (Multiversion Concurrency Control).

40 Serializability (Conflict & View) ▶

A schedule is the order in which operations from multiple concurrent transactions are executed. A schedule is serializable if it produces the same result as if the transactions ran one after another (serially) in some order.

  • Conflict Serializability: Two operations CONFLICT if they are from different transactions, operate on the same data item, and at least one is a WRITE. A schedule is conflict serializable if you can swap non-conflicting operations to make it a serial schedule. We test this using a Precedence Graph.
  • View Serializability: More general (less strict) than conflict. Checks if the initial reads, final writes, and which write each read sees — are all the same as some serial schedule. Harder to check but accepts more schedules.

Every conflict serializable schedule is also view serializable, but not vice versa.

41 Precedence Graph / Conflict Graph ▶

The Precedence Graph (also called Serialization Graph) is used to test if a schedule is conflict serializable.

How to build it:

  • Nodes = each transaction (T1, T2, T3...)
  • Draw a directed edge Ti → Tj if Ti does something BEFORE Tj on the SAME data item AND they conflict (at least one is a Write)
  • Check for Cycles in the graph
If the precedence graph has NO cycle → Schedule is Conflict Serializable ✅
If the precedence graph HAS a cycle → Schedule is NOT Conflict Serializable ❌
The graph is like a "who must come first" rule. If T1 must come before T2, and T2 must come before T1 — that's a cycle, a contradiction, and the schedule is not serializable.
42 Locking — Shared & Exclusive ▶

Locking is the most common concurrency control mechanism. Before accessing data, a transaction must acquire a lock on it.

Lock TypeAlso CalledWho can hold it?Used for?
Shared Lock (S)Read LockMultiple transactions simultaneouslyReading data
Exclusive Lock (X)Write LockOnly ONE transaction at a timeWriting/modifying data

Lock Compatibility Table:

Request →Shared (S)Exclusive (X)
Currently has S✅ Compatible (grant)❌ Not compatible (wait)
Currently has X❌ Not compatible (wait)❌ Not compatible (wait)
Shared lock = a library book that multiple people can read simultaneously. Exclusive lock = a single-user toilet — only one person at a time, no sharing!
43 Two-Phase Locking (2PL) ▶

Two-Phase Locking (2PL) is a protocol that guarantees conflict serializable schedules. A transaction must follow exactly two phases:

  • Phase 1 — Growing Phase: Transaction can only ACQUIRE locks, never release any. The transaction accumulates all the locks it needs.
  • Phase 2 — Shrinking Phase: Transaction can only RELEASE locks, never acquire new ones. Once a lock is released, the transaction enters this phase.
Like packing your bag for a trip: Phase 1 = keep picking up and adding things (locks). Phase 2 = once you take something out (release a lock), you can't add anything new. Only pack, then only unpack.

Variants:

  • Strict 2PL: All exclusive locks held until COMMIT/ABORT. Prevents cascading rollbacks.
  • Rigorous 2PL: ALL locks (shared + exclusive) held until COMMIT/ABORT. Strictest — but avoids all concurrency problems.

2PL guarantees serializability BUT can cause deadlocks.

44 Deadlock in DBMS ▶

A Deadlock occurs when two or more transactions are waiting for each other to release locks — and none of them can proceed. Everyone is stuck waiting!

T1 is holding key to room A and wants room B. T2 is holding key to room B and wants room A. Both are waiting forever. Classic deadlock!
ApproachStrategyHow
PreventionDon't let deadlocks formWait-Die / Wound-Wait schemes using timestamps
AvoidanceCheck before granting locksBanker's Algorithm (check if safe state)
DetectionLet deadlocks happen, then fixWait-for graph — detect cycle → abort one transaction (victim)
TimeoutSimple practical approachIf transaction waits too long → abort it

Wait-for graph: Draw edges T1 → T2 if T1 is waiting for a lock held by T2. If there's a cycle → deadlock detected.

45 Timestamp-Based Protocols ▶

Instead of locks, Timestamp-Based Protocols use timestamps to decide the order of transactions. Each transaction gets a unique timestamp (TS) when it starts. Older timestamps = higher priority.

Each data item tracks two timestamps:

  • Read_TS(X): Timestamp of the most recent transaction that READ X
  • Write_TS(X): Timestamp of the most recent transaction that WROTE X

Thomas Write Rule:

If a transaction tries to write a value that's "in the past" (already overwritten by a newer transaction), simply ignore that write rather than aborting. This allows more transactions to proceed.

It's like newspaper delivery: the morning paper makes the previous day's paper irrelevant. If you're trying to deliver yesterday's news after today's already arrived, just throw it away — don't cause a conflict.
Chapter 7 — Recovery, Storage & Advanced Topics
46 Recovery — Log-Based, Checkpointing ▶

Recovery is the process of restoring the database to a correct state after a failure (system crash, power cut, disk failure, etc.).

Log-Based Recovery

A log (also called a journal) is a file that records every operation before it happens. Each log entry looks like: [Transaction_ID, Data_Item, Old_Value, New_Value]. If a crash happens, we use this log to recover.

Two Strategies:

  • Deferred Update (No-Undo / Redo): Changes are written to disk only AFTER the transaction commits. If it crashes before commit → just ignore the log (no undo needed). After commit → redo from log if not written.
  • Immediate Update (Undo/Redo): Changes may be written to disk during the transaction (before commit). If crash before commit → UNDO using old values. If crash after commit but before disk write → REDO using new values.

Checkpointing

A checkpoint is a snapshot of the database state saved periodically. During recovery, instead of going all the way back to the beginning of the log, we only go back to the last checkpoint. This makes recovery much faster.

47 Rollback & Undo/Redo Operations ▶

After a failure, the recovery manager looks at the log and decides what to do with each transaction:

OperationApplied toWhat it does
UNDOTransactions that STARTED but did NOT commit before crashReverse all their changes using old values from log — restore to state before they started
REDOTransactions that COMMITTED before crash but changes may not have reached diskRe-apply their changes using new values from log — make sure changes are permanent
UNDO = eraser — rub out the mistake. REDO = rewriter — rewrite what was lost. If you committed your homework but your notebook got wet → REDO (rewrite it). If you were halfway through and notebook got wet → UNDO (start fresh, ignore the wet pages).

Cascading Rollback: If T1 reads data written by T2, and T2 fails → T1 must also be rolled back. This is why Strict 2PL is preferred — it prevents reading uncommitted data.

48 Hashing — Static & Dynamic ▶

Hashing is a technique to quickly find where data is stored. A hash function takes a key (like StudentID) and returns a "bucket number" (location) where the record is stored. This allows O(1) average access time!

Imagine 10 lockers numbered 0–9. To find where to put item #47: 47 mod 10 = 7 → put it in locker 7. To find it later: same calculation → locker 7. No searching needed!
Static HashingDynamic Hashing (Extendible)
Number of bucketsFixed at the startGrows/shrinks as needed
ProblemOverflow if too many records; wasted space if too fewMore complex to implement
Collision handlingOverflow chaining / open addressingBucket splitting
Use caseWhen data size is known and fixedWhen data size grows over time

Collision = two keys hash to the same bucket. Handled by chaining (linked list of records in the bucket).

49 File Organization — Heap, Sequential, Hash, Clustered ▶

File Organization refers to how records are physically stored on disk. The choice affects how fast you can search, insert, and delete records.

Not all databases are tables! NoSQL ("Not Only SQL") databases use different data models and are designed for scalability, flexibility, and handling large unstructured data.

51 CAP Theorem ▶

The CAP Theorem (Brewer's Theorem) states that a distributed database system can guarantee at most 2 out of 3 properties at the same time:

PropertyMeaning
C — ConsistencyEvery read gets the most recent write — all nodes see the same data at the same time
A — AvailabilityEvery request gets a response — the system is always up and responding (may not be the latest data)
P — Partition ToleranceThe system continues working even if some network connections between nodes fail (network partition)

In reality, P (Partition Tolerance) is almost always required in distributed systems (networks WILL fail). So the real choice is between:

  • CP systems (Consistent + Partition Tolerant): Sacrifice availability. Return error if can't guarantee latest data. Example: HBase, MongoDB (configured), Zookeeper.
  • AP systems (Available + Partition Tolerant): Sacrifice strict consistency. Always respond but might be slightly stale. Example: Cassandra, DynamoDB, CouchDB.
WhatsApp during a server issue: Availability = you can still open the app. Consistency = you might see old messages. Partition Tolerance = works even with poor network. You can't have all three perfectly at once.
52 Denormalization ▶

Denormalization is the intentional process of adding redundancy back into a database by combining tables — the opposite of normalization. Done to improve READ performance at the cost of some redundancy.

Normalization = putting everything in separate, tidy boxes. Denormalization = keeping some duplicates to save time. Like writing your friend's phone number in multiple places (your phone, your notebook, your PC) so you can find it faster from wherever you are.
  • When to use: When read performance is critical and joins are too slow (e.g., reporting dashboards, OLAP, data warehouses)
  • Downside: Increases storage, data redundancy, and makes updates harder (must update multiple places)
  • Techniques: Pre-joining tables, storing derived/computed values, adding redundant columns

OLAP / Data Warehouse systems often use Star Schema (denormalized) with one big Fact table surrounded by Dimension tables — prioritising fast reads over write efficiency.

53 Query Optimization & Query Processing ▶

When you write an SQL query, the DBMS doesn't just run it blindly. It goes through several steps to find the most efficient way to execute it.

Query Processing Steps:

  • 1. Parsing & Translation: SQL query → internal representation (relational algebra expression)
  • 2. Optimization: Find the cheapest execution plan (considering indexes, joins, data sizes)
  • 3. Evaluation: Execute the chosen plan and return results

What the Query Optimizer does:

  • Choose which index to use (or whether to scan the full table)
  • Choose the best join algorithm (Nested Loop Join, Hash Join, Sort-Merge Join)
  • Choose the order of joins (joining smallest tables first reduces intermediate result size)
  • Apply selection and projection early (push WHERE conditions down the tree — do filters first)
Like a GPS app: you enter a destination (SQL query), the GPS calculates multiple routes (execution plans), picks the fastest one (optimizer), and drives you there (executes it).
54 Cost Estimation in Query Optimization ▶

The query optimizer chooses the cheapest plan by estimating the cost of different execution strategies. Cost is measured primarily in disk I/O (reading blocks from disk is slow).

Cost Factors:

  • Number of disk block accesses — main cost driver
  • CPU time — processing, sorting, hashing
  • Memory usage — buffer pool size available
  • Network communication — in distributed databases

Statistics Used for Estimation:

StatisticMeaning
n_rNumber of tuples (rows) in relation r
b_rNumber of disk blocks containing relation r
l_rSize of each tuple in bytes
f_rBlocking factor — tuples per block
V(A, r)Number of distinct values of attribute A — used to estimate selectivity

Selectivity = what fraction of rows will a condition match? Lower selectivity = fewer rows returned = better to use an index. Higher selectivity = most rows match = full table scan might be better.

If only 2 out of 10,000 students are named "Priya Sharma" — use the name index (very selective). If 9,000 of them are female — skip the gender index and just scan the whole table (not selective enough for an index to help).
